# ğŸ”¹ What is the GIL?

* The **Global Interpreter Lock (GIL)** is a **mutex (lock)** used in the CPython interpreter.
* It ensures that **only one thread executes Python bytecode at a time**, even on multi-core CPUs.
* This makes memory management (reference counting, garbage collection) **safe and simple** in CPython.

---

# ğŸ”¹ Why Does Python Have a GIL?

* Python uses **reference counting** for memory management.
* Without a lock, multiple threads could simultaneously update reference counts â†’ leading to race conditions and memory corruption.
* GIL solves this by **serializing access** to Python objects.

---

# ğŸ”¹ How GIL Affects Multithreading

1. **CPU-bound tasks (bad for GIL)**

   * If threads are doing heavy computation (math, loops, encryption, etc.), GIL prevents them from running in true parallel.
   * Example:

     ```python
     import threading

     def cpu_task():
         x = 0
         for _ in range(10**7):
             x += 1

     threads = [threading.Thread(target=cpu_task) for _ in range(4)]
     for t in threads: t.start()
     for t in threads: t.join()
     ```

     âœ… All threads run **one after another** (no speedup on multi-core CPU).

2. **I/O-bound tasks (good with GIL)**

   * If threads spend time waiting (disk, network, DB), GIL is released during I/O operations.
   * Example: web scraping, API calls.
   * Here, multithreading **does improve performance**.

---

# ğŸ”¹ GIL vs Multiprocessing

* `threading` â†’ affected by GIL (not useful for CPU-bound).
* `multiprocessing` â†’ spawns **separate processes**, each with its own Python interpreter + GIL.
  âœ… True parallel execution on multiple cores.

Example:

```python
    x = 0
    for _ in range(10**7):
        x += 1

for p in procs: p.start()
for p in procs: p.join()
```

This will run in parallel and speed up on multi-core CPUs.

A **race condition** happens when two or more threads or processes try to access and modify a shared resource (like a variable, file, or database record) at the same time, and the programâ€™s outcome depends on the timing of their execution.

ğŸ‘‰ In simple words:
Itâ€™s like two people writing on the same whiteboard at the same timeâ€”if you donâ€™t coordinate, the result will be unpredictable or wrong.

---

### Example in Python

```python
import threading

counter = 0

def increment():
    global counter
    for _ in range(100000):
        counter += 1

t1 = threading.Thread(target=increment)
t2 = threading.Thread(target=increment)

t1.start()
t2.start()
t1.join()
t2.join()

print("Final counter:", counter)
```

ğŸ”¹ You might expect `counter = 200000`, but often youâ€™ll get a smaller number because both threads **race** to update `counter`.
This happens because `counter += 1` is **not atomic** â€” itâ€™s really:

1. Read value
2. Add 1
3. Write back

If two threads interleave at step 1 or 2, updates can be lost.

---

### Where race conditions occur:

* **Multithreading** (shared memory, global variables)
* **Multiprocessing** (shared files, databases)
* **Distributed systems** (multiple servers writing to the same data)
* **Async programming** (concurrent tasks accessing shared state)

---

### How to prevent race conditions:

1. **Locks / Mutex**
   Ensure only one thread modifies a resource at a time.

   ```python
   lock = threading.Lock()
   def increment():
       global counter
       for _ in range(100000):
           with lock:  # only one thread at a time
               counter += 1
   ```

2. **Atomic operations**
   Use thread-safe data structures or atomic variables (like `queue.Queue`, `multiprocessing.Value`, `concurrent.futures`).

3. **Avoid shared state**
   Design systems with immutability, message passing, or partitioned data.

4. **Database transactions**
   Use proper isolation levels to avoid race conditions when multiple clients update data.

# ğŸ”¹ What is `@property` in Python?

`@property` is a **built-in decorator** used in classes to make a **method behave like an attribute**.
It is part of Pythonâ€™s **Descriptor Protocol** and is often used for **encapsulation** (controlling how attributes are accessed, updated, or deleted).

---

## ğŸ”¹ Basic Example

```python
class Student:
    def __init__(self, name, marks):
        self._name = name
        self._marks = marks
    
    @property
    def marks(self):       # Getter
        return self._marks

s = Student("Devendra", 90)
print(s.marks)   # Notice: no parentheses, looks like an attribute
```

âœ… Output:

```
90
```

Even though `marks` is a method, we can access it like an attribute.

---

## ğŸ”¹ Using Setter with `@marks.setter`

```python
class Student:
    def __init__(self, marks):
        self._marks = marks

    @property
    def marks(self):   # Getter
        return self._marks

    @marks.setter
    def marks(self, value):   # Setter
        if value < 0:
            raise ValueError("Marks cannot be negative")
        self._marks = value

s = Student(80)
print(s.marks)   # 80
s.marks = 95     # calls setter
print(s.marks)   # 95
```

---

## ğŸ”¹ Using Deleter with `@marks.deleter`

```python
class Student:
    def __init__(self, marks):
        self._marks = marks

    @property
    def marks(self):
        return self._marks

    @marks.deleter
    def marks(self):
        print("Deleting marks...")
        del self._marks

s = Student(70)
del s.marks
```

---

## ğŸ”¹ Why Use `@property`?

1. **Encapsulation**: Hide internal variables (`_marks`) and control access.
2. **Validation**: Add conditions (e.g., marks canâ€™t be negative).
3. **Read-only Attributes**: If you donâ€™t define a setter, the property becomes read-only.

   ```python
   class Circle:
       def __init__(self, radius):
           self._radius = radius

       @property
       def area(self):
           return 3.14 * self._radius ** 2

   c = Circle(5)
   print(c.area)   # 78.5
   # c.area = 100  âŒ Error (read-only)
   ```

---

## ğŸ”¹ Difference Between `@property` vs Normal Getter/Setter

**Without property (Java-style):**

```python
class Student:
    def __init__(self, marks):
        self._marks = marks
    
    def get_marks(self):
        return self._marks
    
    def set_marks(self, value):
        self._marks = value
```

Usage:

```python

```

**With property (Pythonic):**

```python

    def __init__(self, marks):
        self._marks = marks
    
    @property
    def marks(self):
        return self._marks
    
    @marks.setter
    def marks(self, value):
        self._marks = value
```

Usage:

```python

s.marks = 90
print(s.marks)
```

âœ… Cleaner, more **Pythonic**, no need for `get_` or `set_`.

---

## ğŸ”¹ Quick Summary

* `@property` â†’ getter (read value like an attribute).
* `@<name>.setter` â†’ setter (update value with validation).
* `@<name>.deleter` â†’ deleter (delete the attribute).
* Used for **encapsulation, validation, computed attributes, and read-only attributes**.

## ğŸ”‘ **What is the GIL (Global Interpreter Lock)?**

* The **GIL is a mutex (lock)** used in **CPython** (the default Python implementation).
* It ensures that **only one thread** can execute Python **bytecode** at a time, **even on multi-core processors**.

This means:
ğŸ‘‰ If you have multiple Python threads, only **one thread runs Python code at a time**.
ğŸ‘‰ Other threads must wait until the lock is released.

---

## âš™ï¸ **Why does Python have a GIL?**

* Python uses **reference counting** for memory management (to track objects in memory).
* Reference counting is **not thread-safe** by default.
* To avoid complicated race conditions and make memory management simpler, the **GIL was introduced**.

---

## ğŸ“Œ Example to Understand

```python
import threading

x = 0

def increment():
    global x
    for _ in range(1000000):
        x += 1

t1 = threading.Thread(target=increment)
t2 = threading.Thread(target=increment)

t1.start()
t2.start()
t1.join()
t2.join()

print(x)
```

You might expect `x = 2,000,000`, but because of the GIL, threads donâ€™t really run in parallel â€” they take turns.
So Python doesnâ€™t fully utilize multi-core CPUs for CPU-bound tasks.

---

## âœ… **Key Points for Interview**

1. **GIL restricts true parallelism in threads**

   * Multiple threads can exist, but only one executes at a time.

2. **I/O-bound tasks benefit from threads**

   * When one thread waits for I/O (like network or file), GIL switches to another thread.

3. **CPU-bound tasks donâ€™t benefit from threads**

   * Because threads fight for the GIL, you donâ€™t get real speed-up.

4. **Workarounds**

   * Use **multiprocessing** (each process has its own GIL).
   * Use alternative interpreters (like **Jython, IronPython, or PyPy**) that donâ€™t have GIL.
   * Use **C extensions** that release the GIL during heavy computation (e.g., NumPy).

Nice â€” good question! Here's a clear, practical explanation of how garbage collection (GC) works in Python (mostly CPython), with small examples and tips for debugging/avoiding leaks.

# Overview â€” two parts in CPython

1. **Reference counting (immediate).** Every object has a reference count. When the count drops to 0, the objectâ€™s memory is freed immediately.
2. **Cyclic garbage collector (periodic).** Detects reference cycles (objects that reference each other) that reference counting alone can't reclaim. Itâ€™s a generational, mark-and-sweep style GC for container objects.

# Reference counting â€” simple and deterministic

* When you do `a = obj`, the refcount of `obj` increases. `del a` or reassign reduces the refcount.
* If refcount hits 0, CPython calls the objectâ€™s destructor (`__del__` if present) and frees it right away.
* Fast and predictable, but cycles are a problem (Aâ†’Bâ†’A).

Example showing immediate freeing:

```py
class Foo:
    def __del__(self):
        print("Foo destroyed")

o = Foo()   # refcount 1
o2 = o      # refcount 2
del o
del o2      # refcount -> 0, "Foo destroyed" prints immediately
```

# Cycles and the cyclic GC

* If objects reference each other (A â†’ B â†’ A) and there are no external references, refcounts never reach 0, so CPythonâ€™s cyclic GC finds and collects them.
* The cyclic GC runs periodically and organizes objects into **generations** (0,1,2). New objects go into gen0; surviving objects get promoted. Gen0 is collected often; gen2 rarely.
* The `gc` module exposes control and diagnostic tools.

Example of a cycle:

```py
class Node:
    def __init__(self):
        self.other = None
    def __del__(self):
        print("Node __del__")

a = Node()
b = Node()
a.other = b
b.other = a
del a
del b
# If both have __del__, GC may not collect them automatically (see below).
```

# Important caveat: `__del__` (finalizers) & cycles

* If objects in a cycle define `__del__`, CPython wonâ€™t automatically collect them (it places them in `gc.garbage`) because calling destructors in arbitrary order can be unsafe. This leads to memory not being freed.
* Avoid `__del__` when possible; use context managers or `weakref.finalize` for cleanup.

# The `gc` module â€” inspect & control

```py
import gc

gc.disable()            # disable cyclic GC (rarely needed)
gc.enable()             # enable it

print(gc.get_threshold())   # tuple of thresholds for gen0, gen1, gen2
gc.set_threshold(700, 10, 10)  # tune thresholds

# force a collection and get number of unreachable objects found
unreachable = gc.collect()
print("unreachable objects collected:", unreachable)

# see objects tracked by GC (containers)
print(len(gc.get_objects()))
```

Debugging cycles:

```py
gc.set_debug(gc.DEBUG_LEAK)
gc.collect()
# prints info about unreachable / uncollectable objects
```

# Avoiding memory leaks / best practices

* Prefer simple ownership patterns; avoid unnecessary global references.
* Avoid `__del__` â€” use context managers (`with`) or `weakref.finalize`.
* Break cycles explicitly (set references to `None`) if objects are long-lived.
* Use `weakref` for back-references from childâ†’parent to avoid strong cycles.

Example using `weakref`:

```py
import weakref

class Parent:
    pass

class Child:
    def __init__(self, parent):
        self.parent = weakref.ref(parent)  # does not increment parent's refcount
```

# Other Python implementations

* **CPython:** refcount + cyclic GC (explained above).
* **PyPy:** tracing, generational GC (no refcounting).
* **Jython / IronPython:** use JVM/.NET GC respectively.
  So behavior (timing of collection) can differ across implementations.

# When you should care

* Long-running processes (servers, daemons) â€” leaks accumulate over time.
* Objects holding external resources (file handles, sockets) â€” use explicit close or context managers.
* If memory usage grows unexpectedly, use `gc`, `tracemalloc`, and tools like `objgraph` to diagnose.

# Quick workflow to debug a leak

1. Enable GC debug (`gc.set_debug`).
2. Force `gc.collect()` periodically and log `gc.get_objects()` size.
3. Use `tracemalloc` to see which code allocates memory.
4. Use `objgraph` (third-party) to draw reference chains to a leaking object.
